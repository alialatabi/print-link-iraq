# Matbaty — Customer Ordering UX & Product Audit

**Date:** 2026-07-02
**Method:** 5 parallel read-only analysis agents (ordering journey, auth/onboarding, mobile/Capacitor, performance, feature opportunities). No code was modified.
**Scope:** Customer experience only (web + native app), from landing to delivered order. Admin/designer/reseller out of scope.

---

## The headline insight

`/checkout` is **not a checkout**. It only collects per-item design briefs and creates a `status:'submitted'` order (`CheckoutPage.tsx:157-244`). It never asks for address or payment, and **delivery fee is never shown anywhere in the app** — `lib/orderPricing.ts` has no shipping concept at all (`orderPricing.ts:23-84`). The address is collected in a *separate later session*, only after a designer produces artwork and the customer approves every item (`OrderTracking.tsx:131-134` → `/delivery-address/:orderId`).

The journey is really **two journeys stitched together**: a design-request funnel (home → 4-level catalog → product → cart → "checkout" = design brief) and, days later, a fulfillment funnel (approve → address). Most friction findings below flow from that.

### What's already good (don't regress)

- Android hardware back button fully handled incl. overlay-dismiss-first (`src/lib/native.ts:35-45`)
- Push notifications end-to-end on Android: register + token upsert + tap→order deep link (`src/lib/push.ts`), per-status Arabic copy fired from admin (`AdminPanel.tsx:241-270`), FCM v1 backend (`send-push/index.ts`)
- App Links deep linking with `autoVerify` + `assetlinks.json`; SPA routing via `__appNavigate`
- Safe-area insets, `viewport-fit=cover`, pull-to-refresh, overscroll containment
- Route code-splitting is thorough — every route `React.lazy`; admin/designer/reseller excluded from customer bundle; initial JS/CSS ≈ **244 KB gzipped** (healthy)
- Add-to-cart is instant (synchronous localStorage, `CartContext.tsx:45-57`)
- Al-Waseet regions load per-city (cascading), **not** all 6,265 at once (`useAlwaseetLocations.ts:38-59`)
- Saved-address book exists and works (`AddressPicker.tsx:61-88`, `saved_addresses`)
- Auth is PIN-based: returning users log in with phone + 6-digit PIN (or biometric on native) with **no SMS**; OTP (OTPIQ) only for new accounts / PIN recovery (`send-otp/index.ts:52`)
- OTP inputs: autofocus, paste, numeric keypad, `autoComplete="one-time-code"` (iOS autofill)
- Zero `TODO`/`FIXME`/`HACK` markers in `src/` or `supabase/` — unusually clean codebase

## B. Auth & onboarding

**Current model (verified in code):** returning customer inside the 4-week session sees no auth at all. After expiry / new device: phone → **6-digit PIN** (or fingerprint on native) — no SMS. OTP via OTPIQ fires only for new accounts, no-PIN accounts, or forgot-PIN (`send-otp/index.ts:38-120`, route decision at `:52`).

**New-user first-order gauntlet:** phone → OTP(6) → new PIN(6) + confirm(6) + mandatory consent checkbox (`AuthPage.tsx:355-364`) → forced `/complete-profile` with name\*, province\*, area\*, landmark\* (`CompleteProfile.tsx:66-70`, no skip) → checkout. ≈ 9 fields, 4 network waits, 1 manual SMS read, across 4 full-screen steps. Cart survives the detour (localStorage) — the friction is the steps, not data loss.

| Pri | Recommendation | Evidence | Effort |
|-----|----------------|----------|--------|
| P0 | Stop forcing the address form at signup — collect at the per-order delivery step where it's actually used (checkout never reads profile address) | `CompleteProfile.tsx:66-70`; `AuthPage.tsx:61-67,153`; `CheckoutPage.tsx:131-244` | S–M |
| P0 | Make first-time PIN setup skippable ("set a code to skip SMS next time — or skip"); offer post-order. Skippers just get OTP next login | `AuthPage.tsx:116-158`; `verify-otp/index.ts:109-113` | M |

## E. Feature opportunities (ranked by impact × cheapness)

1. **Order-status updates via WhatsApp/SMS (OTPIQ)** — highest impact, lowest cost. OTPIQ already integrated/funded/rate-limited (`send-otp/index.ts:102-118`); per-status Arabic copy already exists (`AdminPanel.tsx:240-257`, used for FCM); DB status trigger already exists (`notify_customer_on_status_change`). Today a web/idle customer gets **nothing** when their design is ready to approve — which stalls the two-phase flow. One extra OTPIQ call with `smsType:"message"`.
2. **Reorder from past orders** (consolidated P0 #2).
3. **AI-design / saved-design CTA on product pages** — both built but standalone; add prefilled "صمّم بالذكاء الاصطناعي" / "استخدم تصميم محفوظ" on `TemplateDetails`.
4. **Referral/loyalty via existing coupon engine** — `coupons` + `validate-coupon` + `redeem_coupon` RPC + `/my-coupons` all live. Mint "10% off next order" after delivery / referral coupons. **Fix SECURITY_AUDIT M3 first** (customers can't read their coupons).
5. **Finish iOS push** — Android end-to-end done; iOS needs Firebase iOS SDK + APNs key + entitlements (`docs/ios/codemagic-setup.md:69-92`).
6. **Favorites/wishlist** — one `favorites` table + heart toggle on template cards.
7. **Ratings/reviews on delivered orders** → social proof on template pages; complements the `recent_order_activity` ticker.
8. **Share AI designs to WhatsApp/Instagram** with watermark. (Note: `send-to-telegram` is NOT sharing — it's internal print-dispatch: downloads design files, posts to a private Telegram channel, TIF→PNG, flips order to `print_ready`.)
9. **Abandoned-cart nudges** — blocked on server-side cart (cart is localStorage-only); do cart sync first, then reuse #1's OTPIQ plumbing.
10. **Phone-first fingerprint login refinement** — approved plan exists at `docs/superpowers/plans/2026-06-26-phone-first-fingerprint-login.md`, unimplemented.

**Already done (no action):** saved-address book, realtime in-app order tracking with toast+sound (`MyOrders.tsx:72-97`), notification bell, per-service discounts, reseller price overrides, vault reorder.

## Suggested implementation bundles

- **Bundle A — "Express ordering"** (P0 #1–4): checkout resilience + reorder button + optional brief + fee visibility. Directly targets faster/easier ordering.
- **Bundle B — "Feels instant"** (P0 #6–7 + font trim + query caching + search laziness): ~a day of work, transforms perceived speed on mid-range Android.
- **Bundle C — "Bring them back"** (features #1 + #4): OTPIQ order-status messages + referral/next-order coupons.
- **Bundle D — "New-user ramp"** (P0 #5 + Android OTP autofill): collapse first-order onboarding to phone → OTP → checkout.
